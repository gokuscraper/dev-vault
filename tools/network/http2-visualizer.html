HTTP/1.1 Limitations

  • Head-of-line blocking: Only one request at a time per connection. Browser opens 6 connections max per domain.
  • No multiplexing: Response must complete before next request starts on that connection.
  • Header redundancy: Full headers sent with every request (cookies, user-agent, etc.)
  • Domain sharding: Workaround — split assets across subdomains to get more connections.

HTTP/2 Improvements

  • Multiplexing: Multiple streams on a single TCP connection. No head-of-line blocking at HTTP layer.
  • Stream prioritization: Client can hint which resources are more important.
  • HPACK compression: Headers compressed with static table + dynamic table + Huffman encoding.
  • Server Push: Server can proactively send resources the client will need.
  • Binary framing: All communication is in binary frames, not text.

HTTP/2 Stream States

  • idle → open (after HEADERS frame)
  • open → half-closed (after END_STREAM on one side)
  • half-closed → closed (after END_STREAM on other side)
  • Each stream has an odd (client-initiated) or even (server-initiated) ID

HTTP/3 (QUIC)

  • Solves TCP-level head-of-line blocking (HTTP/2 still has this at TCP layer)
  • Built on UDP with QUIC transport protocol
  • 0-RTT connection establishment
  • Independent stream loss recovery — one lost packet doesn't block other streams
