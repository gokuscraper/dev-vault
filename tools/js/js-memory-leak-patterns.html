Common Memory Leak Patterns

  • Closure Leak — Inner function retains reference to outer scope variables, preventing GC even after the outer function returns
  • Detached DOM — DOM nodes removed from the tree but still referenced in JavaScript variables
  • Forgotten Timer — setInterval/setTimeout callbacks hold references that prevent GC
  • Event Listener Leak — Listeners not removed when elements are destroyed, keeping both callback and element alive
  • Global Variable — Accidental globals (missing var/let/const) persist for the page lifetime
  • Circular References — Objects referencing each other, though modern GC handles this (historical issue with IE)

Detection Tools

  • Chrome DevTools → Memory — Heap snapshots, allocation timeline, allocation profiler
  • Heap Snapshot Comparison — Take snapshot → perform action → take snapshot → compare (look for growing delta)
  • Performance Monitor — Chrome DevTools → More tools → Performance monitor → watch "JS Heap size"
  • performance.measureUserAgentSpecificMemory() — Programmatic heap measurement (cross-origin isolated only)

Prevention Checklist

  • Always clearInterval() / clearTimeout() when done
  • Remove event listeners with removeEventListener() or use {once: true}
  • Use WeakRef / WeakMap for caches that shouldn't prevent GC
  • Set references to null when no longer needed
  • Use AbortController to clean up fetch requests and event listeners together
  • In SPAs: clean up in component destroy/unmount lifecycle hooks
