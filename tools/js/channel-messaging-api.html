MessageChannel / MessagePort Basics

A MessageChannel creates two linked MessagePort objects. Messages sent on one port are received on the other. This enables direct, private communication between contexts.

How it Works

const channel = new MessageChannel(); const port1 = channel.port1; const port2 = channel.port2; port1.onmessage = (e) => console.log('Port1 received:', e.data); port2.onmessage = (e) => console.log('Port2 received:', e.data); port1.postMessage('Hello from Port 1'); // Port 2 receives this port2.postMessage('Hello from Port 2'); // Port 1 receives this

Iframe Communication via Ports

MessageChannel enables secure, direct communication with iframes. Transfer one port to the iframe via postMessage, then use the channel for ongoing private communication - no origin checks needed on every message.

The Pattern

// Parent: create channel, send port2 to iframe const channel = new MessageChannel(); iframe.contentWindow.postMessage('init', '*', [channel.port2]); // Parent: listen on port1 channel.port1.onmessage = (e) => { /* handle */ }; // Iframe: receive port, use it directly window.onmessage = (e) => { const port = e.ports[0]; port.onmessage = (e) => { /* handle */ }; port.postMessage('Hello parent!'); };

Port Lifecycle: start, close, transfer

Ports have distinct lifecycle states. Understanding start(), close(), and transfer semantics is crucial for correct usage.

start() behavior

When using addEventListener (not onmessage), you must call port.start() explicitly. The onmessage setter calls start() implicitly.

Start behavior log...

close() behavior

Closing a port prevents further messages. The other port can still send, but messages are silently dropped.

Transfer behavior

Ports are transferable. Once transferred, the original reference becomes neutered (unusable).

Channel Messaging vs BroadcastChannel vs postMessage

Three messaging APIs serve different use cases. Here's how they compare.

Feature MessageChannel BroadcastChannel window.postMessage
Communication 1-to-1 (private) 1-to-many (broadcast) 1-to-1 (window ref)
Cross-origin Yes (via transfer) Same-origin only Yes (with targetOrigin)
Workers Yes (transfer port) Yes (same name) Yes (worker.postMessage)
Transferable Yes No Yes
Lifecycle Explicit (start/close) Explicit (close) Implicit
Security Private channel Same-origin scoped Manual origin check
Use case Dedicated channel between two contexts Pub/sub across tabs/workers Cross-origin window communication
