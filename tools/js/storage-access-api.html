Storage Access API Playground

Manage cross-site cookie access and storage partitioning in third-party contexts

🔍 hasStorageAccess()

Check whether the current document has access to unpartitioned cookies. In a first-party context, this always returns true.

Result
Not checked yet
// Check if we have storage access const hasAccess = await document.hasStorageAccess(); console.log(`Has access: ${hasAccess}`); // In first-party context: always true // In third-party iframe: depends on permission

🔓 requestStorageAccess()

Request access to unpartitioned cookies in a third-party context. Requires user gesture and may show a permission prompt.

Result
Not requested yet
// Must be called with user gesture button.addEventListener('click', async () => { try { await document.requestStorageAccess(); // Access granted - cookies now available const res = await fetch('https://api.example.com/data', { credentials: 'include' }); } catch (err) { // Access denied or dismissed } });

🔗 requestStorageAccessFor()

A top-level document can request storage access on behalf of an embedded origin. This is useful when the first party needs to grant access proactively for its embeds. Only works for Related Website Sets members.

Result
Not requested yet
// Top-level page requests access for embedded origin // Only works within Related Website Sets try { await document.requestStorageAccessFor('https://embedded.example.com'); // Embedded content can now access its cookies } catch (err) { // Not in the same Related Website Set, or denied }

🌐 First-Party vs Third-Party Context

Storage access depends on whether content is loaded in a first-party or third-party context.

First-Party Context
https://mysite.com/page
Cookies: Full access (unpartitioned)
localStorage: Full access
IndexedDB: Full access
Third-Party iframe
<iframe src="https://tracker.com/widget">
Cookies: Blocked (third-party cookie deprecation)
localStorage: Partitioned (per top-level site)
IndexedDB: Partitioned (per top-level site)
Key insight: After requestStorageAccess() is granted, the third-party iframe gains access to its unpartitioned cookies - the same cookies it would have if visited directly as a first-party site. This enables legitimate use cases like embedded login widgets.

🍪 Cookie Partitioning Explained

Modern browsers partition third-party storage by the top-level site, preventing cross-site tracking.

User visits site-a.com

iframe loads tracker.com widget

Partition key: site-a.com
session_id: abc123
pref: dark

User visits site-b.com

iframe loads same tracker.com widget

Partition key: site-b.com
session_id: xyz789
pref: light

Without partitioning, tracker.com would see the same cookies on both sites, enabling cross-site tracking. With partitioning (CHIPS - Cookies Having Independent Partitioned State), each top-level site gets its own cookie jar.

// Set a partitioned cookie (CHIPS) Set-Cookie: session_id=abc123; Secure; SameSite=None; Partitioned; Path=/ // This cookie is isolated per top-level site // site-a.com sees session_id=abc123 // site-b.com sees session_id=xyz789 (different partition) // After requestStorageAccess() is granted: // The iframe gets access to its UNPARTITIONED cookies // (the cookies it has when visited as a first-party)

🔗 Related Website Sets (RWS)

Related Website Sets (formerly First-Party Sets) allow related domains to declare themselves as belonging to the same organization. Members can use requestStorageAccessFor() without user prompts.

Primary: google.com
Members: youtube.com, googleapis.com, gstatic.com, ...
Primary: microsoft.com
Members: live.com, office.com, outlook.com, ...
Primary: example.com
Members: example.co.uk, example-cdn.com, example-auth.com
How to register:
  • Submit a JSON file to the Related Website Sets list on GitHub
  • Each set has one primary domain and up to 5 associated + service domains
  • All domains must be under common ownership
  • A /.well-known/related-website-set.json file must be hosted on each domain
// /.well-known/related-website-set.json (on primary domain) { "primary": "https://example.com", "associatedSites": [ "https://example.co.uk", "https://example-cdn.com" ], "serviceSites": [ "https://example-auth.com" ] } // /.well-known/related-website-set.json (on member domain) { "primary": "https://example.com" }
