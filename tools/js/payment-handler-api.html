⚠ Error Handling Patterns

Common errors and how to handle them in production code.

NotAllowedError
The payment request was not initiated by a user gesture (click, tap, keypress).
Cause: Calling show() without user interaction or calling it twice.
AbortError
The user closed the payment UI or called abort() on the request.
Cause: User dismissed the payment sheet or timeout triggered abort.
InvalidStateError
show() was called on a request that is already showing or has completed.
Cause: Multiple calls to show() on the same PaymentRequest instance.
NotSupportedError
No payment handler found for the specified payment methods.
Cause: Using a payment method URL with no registered handler.
SecurityError
The page is not served over HTTPS or is in a cross-origin iframe without permissions.
Cause: Payment API requires secure context. Check allow attribute on iframes.
TypeError
Invalid arguments passed to PaymentRequest constructor.
Cause: Malformed method data, missing required fields, or invalid amounts.

🔒 Security Considerations

Critical security aspects when implementing payment flows in production.

🛑
HTTPS Required
The Payment Request API only works in secure contexts. All payment pages must be served over HTTPS. No exceptions, even in development (use localhost).
🔒
Never Trust the Client
Always validate payment amounts, currency, and totals on your server. The client-side PaymentRequest is a UI convenience - it does NOT process payments. Your backend must verify everything.
⚠
PCI DSS Compliance
If using basic-card, raw card numbers reach your JavaScript. Use a tokenized payment method (Google Pay, Stripe, etc.) to avoid PCI scope. Token-based methods never expose raw card data.
🕑
Timeout and Abort Handling
Implement timeouts for show() calls. If the user walks away, the payment sheet stays open indefinitely. Use AbortController to cancel after a reasonable timeout (e.g., 5 minutes).
✅
CSP and Permissions Policy
Use Permissions-Policy: payment=(self) to restrict which origins can use the Payment API. In iframes, add allow="payment" attribute.
🔐
Origin Validation
Payment handler URLs must match the origin in the payment method manifest. The browser enforces this, but ensure your server validates the origin of incoming payment confirmations.
