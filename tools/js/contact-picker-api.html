Available Properties

The Contact Picker API supports five properties. Check which ones your browser supports via navigator.contacts.getProperties().

Property Type Description Supported
name string[] Full display names -
email string[] Email addresses -
tel string[] Phone numbers -
address ContactAddress[] Postal addresses -
icon Blob[] Contact profile images -
// Check available properties const properties = await navigator.contacts.getProperties(); console.log('Supported:', properties); // Typically: ["email", "name", "tel"] // On Android Chrome: ["address", "email", "icon", "name", "tel"] // Select contacts with specific properties const contacts = await navigator.contacts.select( ['name', 'email', 'tel'], { multiple: true } ); // Each contact has arrays for each property for (const contact of contacts) { console.log(contact.name[0]); // "John Doe" console.log(contact.email); // ["john@example.com"] console.log(contact.tel); // ["+1-555-0123"] }

Permission Flow Visualization

Unlike other Web APIs, the Contact Picker API does not use the Permissions API. Instead, it requires a user gesture (click/tap) and shows a native contact picker UI.

1

User Gesture Required

The API can only be called in response to a user interaction (click, tap, keypress). Calling it without a gesture throws a SecurityError.

2

Browser Shows Native Picker

The browser displays the device's native contact selection UI. The page has no control over or visibility into this UI.

3

User Selects Contacts

The user chooses which contacts to share. They can select none, one, or many. Only selected contacts are returned - no bulk access.

4

Data Returned to Page

Only the requested properties of selected contacts are returned. The promise resolves with an array of contact objects. If the user cancels, an empty array is returned.

5

No Persistent Access

Each call to select() requires a new user gesture. There's no way to get persistent access or monitor contact changes. This is a one-shot, user-controlled operation.

Error Scenarios

SecurityError - Called without user gesture, or browsing context not secure (not HTTPS)
TypeError - No properties specified, or invalid property names
InvalidStateError - Contact picker is already showing (only one at a time)

Progressive Enhancement Pattern

// Feature detection + fallback async function getContactInfo() { // Check if Contact Picker is available if ('contacts' in navigator && 'ContactsManager' in window) { try { const props = await navigator.contacts.getProperties(); const contacts = await navigator.contacts.select( props, { multiple: false } ); if (contacts.length > 0) { return fillFormFromContact(contacts[0]); } } catch (e) { console.log('Contact Picker error:', e); } } // Fallback: show manual form showManualEntryForm(); } function fillFormFromContact(contact) { document.getElementById('name').value = contact.name?.[0] || ''; document.getElementById('email').value = contact.email?.[0] || ''; document.getElementById('phone').value = contact.tel?.[0] || ''; }

Privacy Considerations

The Contact Picker API was designed with privacy as a primary concern. Understanding these design decisions helps explain why the API works the way it does.

  • 🛡️

    User-Initiated Only

    The API requires a user gesture (click/tap). No background access, no silent reads. The user must actively choose to share contacts each time.

  • 🔒

    No Bulk Access

    Unlike native apps that can request full contact list access, the Contact Picker only returns explicitly selected contacts. Sites cannot enumerate or crawl the address book.

  • 🔍

    Selective Properties

    Sites must declare which properties they need. The browser only returns requested fields. If a site only needs names, it cannot access email or phone numbers.

  • ⚡

    One-Shot Design

    No persistent permission is granted. Each call to select() shows the native picker again. There's no way to "remember" access or monitor contact changes.

  • 🌐

    HTTPS Required

    The API is only available in secure contexts (HTTPS). This prevents man-in-the-middle attacks from intercepting contact data.

  • 📱

    Native UI Control

    The contact picker UI is controlled by the browser/OS, not the web page. The site cannot customize, overlay, or manipulate the picker. This prevents deceptive UI patterns.

Developer Best Practices

  • • Always provide a manual entry fallback for unsupported browsers
  • • Only request properties you actually need
  • • Explain why you need contact access before triggering the picker
  • • Don't store contact data longer than necessary
  • • Never send raw contact data to third-party services without consent
  • • Handle the empty array case (user cancelled) gracefully
