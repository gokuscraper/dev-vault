How DNS Resolution Works

DNS (Domain Name System) translates human-readable domain names into IP addresses. The process involves multiple servers working together in a hierarchical system.

Resolution Steps

  • Browser Cache — Browser checks its own DNS cache first (Chrome: chrome://net-internals/#dns)
  • OS Cache — Operating system's DNS resolver cache (stub resolver)
  • Recursive Resolver — ISP or configured DNS server (e.g., 8.8.8.8). Does the heavy lifting
  • Root Server — 13 root server clusters (a.root-servers.net through m.root-servers.net). Returns TLD nameserver
  • TLD Server — Top-Level Domain server (.com, .org, .net). Returns authoritative nameserver
  • Authoritative NS — The domain's own nameserver. Returns the actual DNS record

Record Types

A IPv4 address AAAA IPv6 address CNAME Canonical name (alias) MX Mail exchange NS Nameserver TXT Text record (SPF, DKIM, etc.) SOA Start of Authority PTR Reverse lookup

Key Concepts

  • TTL (Time To Live) — How long a record can be cached (seconds). Low TTL = fresher data but more queries
  • Recursive vs Iterative — Recursive: resolver does all work. Iterative: each server refers to the next
  • DNSSEC — Cryptographic signatures that verify DNS responses haven't been tampered with
  • DNS over HTTPS (DoH) — Encrypts DNS queries over HTTPS (port 443). Used by browsers
  • DNS over TLS (DoT) — Encrypts DNS queries over TLS (port 853). Used by OS resolvers
  • Anycast — Root/TLD servers use anycast routing — same IP, nearest physical server responds
