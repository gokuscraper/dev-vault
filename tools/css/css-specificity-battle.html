The Cascade Order

When two declarations compete, the browser resolves them in this order (highest priority first):

#Origin & ImportanceExample
1Transition declarationsActive CSS transitions
2User agent !importantBrowser defaults with !important
3User !importantUser stylesheet with !important
4Author !importantcolor: red !important;
5Animation declarationsActive @keyframes
6Author normalYour regular CSS
7User normalUser stylesheet
8User agent normalBrowser defaults

Within the same origin & importance, specificity determines the winner. If specificity is equal, source order (last one wins).

@layer Impact

Cascade layers (@layer) sit between origins. Unlayered styles beat all layers. Later-declared layers beat earlier ones. !important reverses layer order.

Common Gotchas

:is() and :not() inherit highest specificity

:is(.foo, #bar) has specificity of #bar (1,0,0) even when matching .foo. Use :where() for zero specificity.

:where() always has zero specificity

:where(#id .class) = (0,0,0). Useful for default styles meant to be easily overridden.

Inline styles beat everything (except !important)

style="color:red" = (1,0,0,0). Only !important in a stylesheet or another inline !important can override.

!important is not about specificity

It changes the origin in the cascade (moves to "author important" level). Two !important declarations still compare by specificity.

Universal selector * has zero specificity

*, +, ~, > combinators add nothing to specificity.

Repeated classes increase specificity

.foo.foo = (0,2,0) and beats .foo = (0,1,0). A valid but ugly override technique.

::pseudo-elements are type selectors

::before, ::after, ::placeholder count as element selectors (0,0,1), not class selectors.

:nth-child() counts as a pseudo-class

:nth-child(2) = (0,1,0). Same as any class or attribute selector.
