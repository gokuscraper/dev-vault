Nesting with & Selector

The & represents the parent selector in nested CSS. It's required for pseudo-classes, pseudo-elements, and compound selectors, but optional for descendant selectors (as of the relaxed syntax).

Interactive & Patterns live editor

& Rules:

  • & = the full parent selector, not just the element name
  • &:hover = pseudo-class on parent (& is required)
  • &::before = pseudo-element on parent
  • & .child = descendant (& is optional since relaxed syntax)
  • .parent & = reverse nesting — & is the child
  • &.modifier = compound selector (no space!)

The &-concatenation Trap migration pitfall

SCSS (works)
.card { &-title { font-size: 18px; } &-body { font-size: 14px; } &--featured { border-color: gold; } } /* Outputs: .card-title { ... } .card-body { ... } .card--featured { ... } */
Native CSS (does NOT work)
.card { &-title { font-size: 18px; } /* INVALID: browsers don't concatenate strings */ } /* Use data attributes or separate classes instead: */ .card { & .card-title { font-size: 18px; } & .card-body { font-size: 14px; } &.card--featured { border-color: gold; } }

Performance Impact

Does nesting CSS rules affect performance? The browser unrolls nested rules into flat selectors internally. Deep nesting can increase selector complexity and matching time.

Best Practices

  • Keep nesting to 3 levels maximum
  • Use class selectors over type selectors
  • Avoid nesting inside @layer inside nesting
  • Profile with DevTools CSS Overview

Anti-Patterns

  • Nesting 5+ levels deep
  • Nesting universal selectors (*)
  • Deeply nested attribute selectors
  • Matching DOM structure 1:1
