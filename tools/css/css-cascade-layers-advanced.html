@layer Ordering Rules

Layer order is determined by the first appearance of each layer name. Later layers have higher priority. Unlayered styles always win over layered styles (unless !important is involved).

Layer Priority Visualization
1 reset lowest
2 base
3 components
4 utilities
5 unlayered highest
CSS
/* Order declaration — first appearance wins */ @layer reset, base, components, utilities; @layer reset { * { margin: 0; padding: 0; } } @layer base { body { font-family: sans-serif; } } @layer components { .btn { padding: 0.5rem 1rem; } } @layer utilities { .text-center { text-align: center; } } /* Unlayered styles beat ALL layers */ .special { color: red; }

Nested Layers (outer.inner)

Layers can be nested. The flat name framework.base is equivalent to nesting base inside framework. Nested layers follow the same ordering rules within their parent.

/* Nested layer syntax */ @layer framework { @layer base { p { color: blue; } } @layer theme { p { color: green; } } } /* Equivalent flat syntax */ @layer framework.base { p { color: blue; } } @layer framework.theme { p { color: green; } } /* framework.theme wins over framework.base because theme appears after base */
Nested Layer Hierarchy
1 framework
1.1 framework.base
1.2 framework.theme
2 app
2.1 app.components
2.2 app.utilities
Key Rule: All sublayers of framework have lower priority than any sublayer of app, regardless of specificity.

Layer Priority vs Specificity

Layer order completely overrides specificity. A simple selector in a higher-priority layer beats a complex selector in a lower-priority layer. This is a fundamental shift in how CSS conflicts are resolved.

Critical Concept: Cascade layers are evaluated BEFORE specificity. A .class in a later layer beats #id.class[attr]:hover in an earlier layer.
@layer base, override; @layer base { /* Specificity: (1,1,0) — very high */ #myElement.special { color: blue; font-weight: bold; } } @layer override { /* Specificity: (0,1,0) — lower */ .special { color: red; /* WINS because layer order > specificity */ } }
Cascade Resolution Order
1. Origin & Importance (user-agent, user, author)
2. Context (shadow DOM)
3. Layer order ← @layer fits here
4. Specificity
5. Source order

!important Reversal in Layers

When !important is used inside layers, the priority order is reversed. Lower-priority layers' !important declarations win over higher-priority layers' !important declarations. This mirrors how the cascade handles importance across origins.

Mind-bending rule: With !important, the FIRST layer in the order has the HIGHEST priority. This is the opposite of normal (non-important) declarations.
Normal order (no !important)
1 reset lowest
2 base
3 components
4 utilities highest
!important order (REVERSED)
1 reset highest !
2 base
3 components
4 utilities lowest !
@layer reset, base, theme; @layer reset { p { color: black !important; } /* !important from lowest layer = HIGHEST priority */ } @layer theme { p { color: purple !important; } /* !important from highest layer = LOWEST priority */ } /* Result: color is BLACK (reset wins with !important) */
Live Demo

This text is BLACK because reset's !important wins (reversed order).

Why this exists: It ensures that reset/normalize layers can use !important to truly lock down base styles, even if later layers also use !important. The earlier (more fundamental) layer wins.

Anonymous Layers

Layers without a name are anonymous. Each anonymous @layer { } block creates a separate, unique layer. They cannot be referenced or appended to later.

/* Each anonymous block is a separate layer */ @layer { p { color: red; } } @layer { p { color: blue; } /* This is a DIFFERENT anonymous layer */ /* It has higher priority (appears later) */ } /* You cannot add more rules to either anonymous layer. They are sealed at definition. */ /* Use case: isolating third-party CSS */ @layer { /* All of widget.css goes here — isolated! */ .widget { /* ... */ } } /* Named layers CAN be extended */ @layer theme { p { font-size: 16px; } } @layer theme { p { line-height: 1.5; } } /* appended */

@import with layer()

You can assign imported stylesheets to layers using @import url() layer(name). This is ideal for controlling third-party CSS priority.

/* Import into a named layer */ @import url("reset.css") layer(reset); @import url("bootstrap.css") layer(framework); @import url("theme.css") layer(theme); /* Pre-declare order before imports */ @layer reset, framework, theme, app; @import url("reset.css") layer(reset); @import url("bootstrap.css") layer(framework); /* Import into anonymous layer */ @import url("legacy-widget.css") layer; /* Layer order with @import: The @layer declaration establishes order. @import layer() respects that order. */
Browser Support Note: @import layer() is supported in all modern browsers (Chrome 99+, Firefox 97+, Safari 15.4+). This is the cleanest way to layer third-party CSS.
/* Real-world example: layering a framework */ /* 1. Declare layer order upfront */ @layer normalize, framework, app, utilities; /* 2. Import framework into its layer */ @import url("https://cdn.example.com/framework.css") layer(framework); /* 3. Your app styles — higher priority than framework */ @layer app { .card { /* Overrides framework .card without !important and without specificity wars */ border-radius: 12px; } } /* 4. Utility classes — highest layered priority */ @layer utilities { .hidden { display: none; } }

revert-layer Keyword

revert-layer rolls a property back to the value it would have in the previous layer. It's like revert but for layers instead of origins.

@layer base, theme, overrides; @layer base { .btn { color: gray; background: white; } } @layer theme { .btn { color: blue; background: lightblue; } } @layer overrides { .btn { /* Keep theme's background, but revert color to base layer */ color: revert-layer; /* Result: color = blue (from theme) */ /* Wait, no — revert-layer reverts to the PREVIOUS cascade layer */ } } /* Comparison: revert-layer → previous layer's value revert → previous origin's value unset → inherited or initial initial → spec initial value */
Revert Keyword Comparison
Keyword Reverts to
initial Spec-defined initial value
inherit Parent element's value
unset inherit or initial (depends)
revert Previous cascade origin
revert-layer Previous cascade layer

Migration Strategy

Step-by-step guide for migrating existing CSS to use cascade layers without breaking anything.

/* STEP 1: Wrap ALL existing CSS in a single layer */ /* This changes nothing — one layer behaves like no layers */ @layer legacy { /* paste all existing CSS here */ .header { color: navy; } .card { border: 1px solid gray; } /* ... */ } /* STEP 2: New CSS goes outside (unlayered = higher priority) */ .header { color: teal; } /* wins over legacy */
/* STEP 3: Split legacy into logical layers */ @layer reset, legacy, components, utilities; @layer reset { * { margin: 0; box-sizing: border-box; } } @layer legacy { /* Gradually move code out of here */ .old-widget { /* ... */ } } @layer components { .card { border-radius: 8px; } .btn { padding: 0.5rem 1rem; } } @layer utilities { .sr-only { /* screen reader only */ } }
/* STEP 4: Layer third-party CSS */ @layer reset, vendor, legacy, components, utilities; @import url("bootstrap.css") layer(vendor); /* Now Bootstrap can NEVER override your component styles, regardless of specificity. No more !important wars. */
Migration Tip: Start with Step 1 (wrap everything in one layer). This is zero-risk and lets you start writing new styles that automatically override old ones.
Watch out: Unlayered !important declarations have LOWER priority than layered !important. If your existing CSS uses !important, moving it into a layer changes behavior.
