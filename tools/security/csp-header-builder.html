What is CSP?

Content Security Policy is an HTTP header that tells browsers which resources are allowed to load. It's the primary defense against XSS (Cross-Site Scripting) attacks.

Key Source Values

  • 'self' — Same origin only (protocol + host + port)
  • 'none' — Block everything for this directive
  • 'unsafe-inline' — Allow inline scripts/styles (weakens XSS protection significantly)
  • 'unsafe-eval' — Allow eval(), Function(), setTimeout(string) (dangerous)
  • 'strict-dynamic' — Trust scripts loaded by already-trusted scripts (nonce-based)
  • 'nonce-{base64}' — Allow specific inline script/style with matching nonce
  • 'sha256-{hash}' — Allow specific inline script/style by hash
  • https: — Any HTTPS URL
  • data: — data: URIs (images, fonts)
  • blob: — blob: URIs
  • *.example.com — Wildcard subdomain matching

Strictness Levels

  • Weak — Uses 'unsafe-inline' or 'unsafe-eval', or missing critical directives
  • Medium — Has default-src, avoids 'unsafe-*', but may allow broad sources
  • Strict — Nonce/hash-based, no 'unsafe-*', specific sources, uses 'strict-dynamic'

Common Mistakes

  • Using 'unsafe-inline' — Defeats XSS protection. Use nonces or hashes instead.
  • Overly broad wildcards — *.googleapis.com may include user-controlled subdomains
  • Missing default-src — Without fallback, unspecified resource types are unrestricted
  • Forgetting frame-ancestors — Needed to prevent clickjacking (replaces X-Frame-Options)
  • Not using report-uri — Deploy in report-only mode first to avoid breaking your site

Deployment Strategy

  • Step 1: Use Content-Security-Policy-Report-Only header first
  • Step 2: Set up report-uri to collect violations (use report-uri.com or Sentry)
  • Step 3: Fix violations in your code
  • Step 4: Switch to enforcing Content-Security-Policy
