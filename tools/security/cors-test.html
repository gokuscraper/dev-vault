What is CORS?

Cross-Origin Resource Sharing (CORS) is a security mechanism that controls which websites can access resources from another domain. When your frontend at app.example.com tries to fetch data from api.another.com, the browser checks CORS headers to decide whether to allow it.

How it works

Simple requests (GET/POST with standard headers) go directly. The browser checks the response for Access-Control-Allow-Origin.

Preflighted requests (PUT/DELETE, custom headers, non-standard content types) trigger an OPTIONS request first. The server must respond with allowed methods and headers before the actual request proceeds.

Credentialed requests (cookies, auth headers) require Access-Control-Allow-Credentials: true and a specific origin (not *).

Key CORS Headers

Access-Control-Allow-Origin — Which origins can access. * = any origin.

Access-Control-Allow-Methods — Allowed HTTP methods (GET, POST, PUT, etc.).

Access-Control-Allow-Headers — Allowed custom request headers.

Access-Control-Max-Age — How long (seconds) to cache preflight results.

Access-Control-Expose-Headers — Which response headers JS can read.

Access-Control-Allow-Credentials — Whether cookies/auth can be sent.

Common CORS Errors

"No 'Access-Control-Allow-Origin' header" — The server doesn't include CORS headers. Fix: add the header server-side.

"Wildcard with credentials" — Can't use * with credentials: include. Fix: echo the specific origin.

"Method not allowed" — The preflight didn't include the method. Fix: add it to Access-Control-Allow-Methods.

"Request header not allowed" — Custom header not in Access-Control-Allow-Headers. Fix: add it.

Server-Side Fix Examples

Express.js:

app.use(cors({ origin: 'https://myapp.com', credentials: true }))

Nginx:

add_header Access-Control-Allow-Origin "https://myapp.com";

Apache (.htaccess):

Header set Access-Control-Allow-Origin "https://myapp.com"
